# 用户测试计划

## 目标

下学期用户测试的目标不是证明模型本身最聪明，而是验证：

```text
用户是否能理解并使用 Builder 生成/运行一个 Workflow，
以及模板化 Workflow 是否能降低 Agent 制作和配置成本。
```

## 测试对象

建议至少做 2 轮测试。

| 轮次 | 人数 | 对象 | 目的 |
|---|---:|---|---|
| Pilot Test | 2-3 | 组内/熟人 | 检查流程、问题表、Demo 是否能跑 |
| User Test 1 | 6-8 | 非本组同学 | 发现可用性问题 |
| User Test 2 | 6-8 | 非本组同学或上一轮部分用户 | 验证修改后是否变好 |
| Exhibition Rehearsal | 2-3 | 完全没看过项目的人 | 检查 2 分钟 Demo 是否讲得清楚 |

## 测试系统

当前系统定位为：

```text
Template-based Multi-Agent Workflow Builder MVP
```

测试时不要说“这是完整产品”。应说明：

```text
这是一个用于验证 Workflow Template 可复用性的 MVP。
用户可以选择模板、输入任务、运行 Workflow，并查看节点 Trace 和最终结果。
```

## 核心测试问题

### Q1. 用户能否理解系统在做什么？

观察点：

- 用户是否知道这是 Builder，不只是聊天机器人
- 用户是否能理解 Template / Workflow / Node / Trace
- 用户是否能解释“为什么有多个节点”

### Q2. 用户能否完成指定任务？

观察点：

- 是否能选择正确模板
- 是否能输入任务
- 是否能运行 Workflow
- 是否能看懂最终输出
- 是否能看懂 Trace

### Q3. Builder 是否比手动配置更容易？

观察点：

- 用户觉得模板选择是否比从零配置更简单
- 用户是否理解“同一结构可以换业务场景”
- 用户是否能说出复用价值

### Q4. 推荐结果是否有用？

观察点：

- 输出是否符合用户输入
- 是否使用了天气/偏好/购物记录等信息
- 是否有推荐理由和优先级
- 用户是否愿意继续使用

## 测试任务

### Task A: 穿搭推荐 Workflow

给用户的任务：

```text
请选择穿搭推荐模板，并输入：
“下周去青岛旅游，帮我推荐适合的衣服。”
```

记录：

- 是否成功完成
- 花费时间
- 用户是否看懂推荐理由
- 用户是否注意到 Question / Weather / Shopping / Recommendation 节点

### Task B: 通勤穿搭 Workflow

给用户的任务：

```text
请选择通勤穿搭模板，并输入：
“明天首尔上班，帮我推荐正式一点的穿搭。”
```

记录：

- 是否成功完成
- 用户是否能区分它和 Task A 的不同
- 用户是否理解模板复用

### Task C: 非推荐领域 Workflow

给用户的任务：

```text
请选择发表规划或客服工单模板，运行一次，并说明这个模板和穿搭模板的共同点。
```

目的：

验证用户是否理解：

```text
本项目不是单一推荐 App，而是通用 Workflow Builder MVP。
```

## 量化指标

| 指标 | 记录方式 | 目标 |
|---|---|---|
| Task Success Rate | 成功/失败 | 80% 以上 |
| Time on Task | 秒 | 每个任务 2 分钟内 |
| User Confusion Count | 观察者记录 | 每人 3 次以内 |
| Workflow Understanding Score | 1-5 分 | 平均 4 分以上 |
| Output Usefulness Score | 1-5 分 | 平均 4 分以上 |
| Builder Value Score | 1-5 分 | 平均 4 分以上 |

## 定性记录

每个用户至少记录 3 类反馈：

1. 用户看不懂的地方
2. 用户觉得有用的地方
3. 用户希望增加或修改的地方

## 测试流程

每个用户控制在 15-20 分钟。

1. 简短说明项目背景，1 分钟
2. 让用户完成 Task A，3-5 分钟
3. 让用户完成 Task B，3-5 分钟
4. 让用户完成 Task C，3-5 分钟
5. 填写问卷，3 分钟
6. 简短访谈，3 分钟

## 角色分工

| 角色 | 负责内容 |
|---|---|
| 主持人 | 读测试脚本，不提示答案 |
| 观察者 | 记录时间、错误、卡住的位置 |
| 技术负责人 | 处理程序启动和异常，不参与引导 |
| 记录负责人 | 汇总问卷和访谈结果 |

## PPT 中的呈现方式

最终发表建议放一页：

```text
User Test Result
```

内容包括：

- 参与人数
- 完成率
- 平均完成时间
- 用户理解评分
- 推荐结果有用性评分
- 代表性反馈 2-3 条
- 修改前后对比

## 注意事项

测试时不要替用户操作，也不要解释每一步该点哪里。  
如果用户卡住，先记录卡住的位置，再给最小提示。

如果用户说“不知道这是什么”，这不是失败，而是重要结果。  
这说明 UI 或发表说明里需要更清楚地区分：

```text
Chatbot
Agent
Workflow
Builder
```
