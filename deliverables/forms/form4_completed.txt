1. 요구기능 설명
본 요구기능은 사용자의 자연어 의류 추천 요청을 분석하고, 도시/날짜/목적/스타일 정보가 부족한 경우 추가 질문을 수행한 뒤, 외부 날씨 API와 로컬 쇼핑 이력 데이터를 결합하여 개인화된 착장 추천 결과를 생성한다. 시스템은 JSON 기반 Workflow 정의를 읽어 6개 Node를 순차 실행하며, 데스크톱 UI와 웹 UI에서 실행 결과와 Trace를 확인할 수 있다.
1.1 주요 상세기능 설명
상세기능 ID	상세기능명	정의	상세 설명
NENE_OUTFIT_001_FD_001	Request Parser Node	사용자 요청 분석	입력 문장에서 도시, 날짜, 사용자 ID, 의도 키워드를 추출한다.
NENE_OUTFIT_001_FD_002	Question Node	부족 정보 확인	도시, 날짜, 목적/스타일 중 누락된 필드를 탐지하고 순차 추가 질문을 생성한다.
NENE_OUTFIT_001_FD_003	Weather Tool Node	날씨 조회	Open-Meteo Geocoding API와 Forecast API를 호출하여 일별 날씨를 표준 데이터로 변환한다.
NENE_OUTFIT_001_FD_004	Shopping History Analysis Node	구매 이력 분석	사용자별 구매 기록에서 선호 스타일, 색상, 카테고리, 대표 아이템을 산출한다.

NENE_OUTFIT_001_FD_006	Compose Node	최종 응답 구성	output_template에 컨텍스트 값을 주입하여 최종 사용자 응답을 렌더링한다.
1.2 요구기능 동작 절차
사용자가 데스크톱 UI 또는 웹 UI에 자연어 추천 요청을 입력한다.
Request Parser Node가 도시명, 날짜 표현, 사용자 ID, 의도 키워드를 분석한다.
Question Node가 누락 필드를 확인한다. 정보가 부족하면 해당 질문만 응답으로 반환하고 Workflow를 중단한다.
정보가 충분하면 Weather Tool Node가 도시 좌표와 일별 예보 데이터를 조회한다.
Shopping History Analysis Node가 shopping_history.json을 읽고 사용자의 선호 스타일/색상/카테고리를 집계한다.
Recommendation Node가 온도 구간 규칙(cold/mild/warm)과 조건부 추가 아이템(우산, 얇은 겉옷)을 적용한다.
Compose Node가 최종 추천문과 추천 순위, 실행 Trace를 UI에 표시한다.
1.3 동작 규칙
Workflow 실행 순서는 request_parser -> question -> weather -> shopping_analysis -> recommendation -> compose로 고정한다.
도시, 날짜, 목적/스타일 정보 중 하나라도 부족하면 Question Node가 우선 실행되고 외부 API 호출은 수행하지 않는다.
도시 별칭은 서울, 부산, 인천, 도쿄, 오사카, 뉴욕, 칭다오, 베이징, 상하이 등을 지원한다.
평균기온 8도 이하에는 보온성, 8도 초과 22도 이하에는 레이어드, 22도 이상에는 통풍성 중심 추천을 적용한다.
강수확률이 50% 이상이면 우산을 추가 추천하고, 일교차가 10도 이상이면 얇은 겉옷을 추가 추천한다.
1.4 가정(Assumptions)
사용자별 구매 이력은 data/shopping_history.json에 등록되어 있어야 한다.
날씨 조회를 위해 Open-Meteo API에 접근 가능한 네트워크 환경이 필요하다.
MVP 범위에서는 실제 쇼핑몰 OAuth, 상품 구매 API, 드래그앤드롭 Builder UI는 제외한다.
추천 결과는 데모용 규칙 기반 결과이며, 추후 룰 확장 또는 ML 기반 추천으로 확장 가능하다.
2. 기술설계 #1: MultiAgentWorkflowEngine
2.1 기능 설계
Python 객체지향 구조로 구현한다. MultiAgentWorkflowEngine은 Workflow 설정 파일을 로드하고, AgentBuilderEngine을 기반 실행 엔진으로 사용하여 각 Node를 순차적으로 실행한다. 각 단계의 실행 결과는 context 딕셔너리에 누적되며 TraceStep 리스트로 UI에 전달된다.
선행조건(Pre-condition)
configs/outfit_workflow.json과 configs/outfit_agent.json이 정상 JSON 형식이어야 한다.
data/shopping_history.json에 요청 user_id가 존재해야 한다.
WeatherTool이 도시 좌표 및 예보 데이터를 반환할 수 있어야 한다.
상세기능 절차
run(user_message, user_id)에서 query와 selected_user를 결정한다.
_run_request_parser()가 base_engine._plan()을 호출하여 city_display, city_query, day_offset, matched_keywords를 생성한다.
_run_question_agent()가 missing_fields를 계산한다.
needs_clarification이 True이면 clarification_message를 answer로 하여 _finalize()한다.
_run_weather_agent()가 WeatherTool.get_daily_weather() 결과를 weather_summary로 저장한다.
_run_shopping_analysis_agent()가 analyze_shopping_history() 결과를 context에 저장한다.
base_engine._decide()가 온도 규칙과 보유 아이템 기반 추천 순위를 계산한다.
base_engine._render()가 output_template을 최종 응답으로 렌더링한다.
후행조건(Post-condition)
WorkflowResult.answer에 사용자에게 보여줄 최종 응답 또는 추가 질문이 저장된다.
WorkflowResult.trace에 각 Node 이름, 설명, 데이터가 저장되어 UI Trace 영역에 표시된다.
WorkflowResult.context에 workflow_name, workflow_agents, workflow_execution 메타데이터가 포함된다.

3. 기술설계 #2: UI 설계
3.1 입력 UI
데스크톱 UI: 사용자 선택 콤보박스, 사용자 입력/추가 답변 텍스트 영역, Run Workflow 버튼, 예시 입력 버튼, 추적 컨텍스트 표시 영역으로 구성한다.
웹 UI: 한국어/중국어 언어 토글, Demo Config 탭, 쇼핑 기록 사용자 선택, 예시 입력 칩, Query textarea, 실행 버튼으로 구성한다.
정보 부족 시 UI는 기존 query를 conversation_context로 보관하고 다음 입력을 보충 답변으로 병합한다.
3.2 출력 결과
Workflow Output 영역에는 최종 추천문 또는 추가 질문 메시지를 표시한다.
Workflow Trace 영역에는 Node 번호, Node 이름, 실행 상세, 단계별 데이터 요약을 표시한다.
Technical Details에는 Loaded Config, Shopping History Analysis, Raw Trace JSON을 접을 수 있는 영역으로 제공한다.
4. 기타사항
테스트 실행 명령은 tools/dev/run_tests.cmd이며 README 기준 기대 결과는 28 tests OK이다.
현재 MVP 범위는 JSON Workflow, 순차 실행, 날씨 도구 호출, 구매 이력 분석, 규칙 기반 추천, 웹/데스크톱 UI, 테스트이다.
향후 확장 범위는 시각적 Builder UI, 실제 쇼핑 플랫폼 API, 추천 규칙 확장, 복잡한 재시도/분기/모니터링 기능이다.
